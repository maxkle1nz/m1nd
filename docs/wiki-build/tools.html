Tool Reference

Complete reference for all 63 MCP tools. Each tool is mapped to a semantic domain defined by the five connectome glyphs.

63 tools across 8 groups
⍌
Signal
Activation, flow, propagation. How information spreads through the graph.
4 tools
⍌ activate — Spreading activation query

Find related nodes by propagating signal through the graph. Starts from a query, activates matching nodes, then spreads energy along weighted edges. Returns ranked results by activation strength. The core discovery mechanism of the connectome.

Parameters
NameTypeDescription
querystringWhat to search for. Can be a concept, module name, or natural language question.
depth1-5How many hops to propagate. Higher = broader but noisier results.
threshold0.0-1.0Minimum activation energy to include in results. Higher = more precise.
agent_idstringCalling agent identifier for multi-agent coordination.
Example
// Find everything related to authentication m1nd.activate({ "query": "authentication middleware", "depth": 3, "threshold": 0.4, "agent_id": "jimi" })
Exploration Research Impact scoping Onboarding
⍌ ingest — Load codebase into graph

Parse a codebase or JSON data and build the connectome graph. Extracts modules, functions, types, and files as nodes. Discovers imports, calls, contains, and co-change relationships as edges. Supports full rebuild or incremental updates.

Parameters
NameTypeDescription
sourcepathPath to the codebase directory or JSON file to ingest.
modefull | incrementalFull rebuilds the entire graph. Incremental updates only changed files.
agent_idstringCalling agent identifier.
Example
// Ingest a Rust project m1nd.ingest({ "source": "./my-project", "mode": "full", "agent_id": "jimi" })
Session start After code changes Codebase onboarding
⍌ learn — Hebbian feedback

Strengthen or weaken graph connections based on usage feedback. When you use activate and the results are helpful, call learn with "correct". If results were off, use "wrong". The graph adapts over time, improving result quality through Hebbian plasticity.

Parameters
NameTypeDescription
querystringThe original query that produced results.
feedbackcorrect | wrong | partialWas the result useful? Adjusts edge weights accordingly.
agent_idstringCalling agent identifier.
Example
// The activation results were on point m1nd.learn({ "query": "authentication middleware", "feedback": "correct", "agent_id": "jimi" })
After every activate Graph tuning Quality improvement
⍌ warmup — Prime graph for a task

Pre-activate a relevant subgraph before starting focused work. Primes the spreading activation network so subsequent queries in the same domain return faster, more relevant results. Think of it as "warming up the neural pathways."

Parameters
NameTypeDescription
contextstringDescription of the task you are about to work on.
agent_idstringCalling agent identifier.
Example
// About to refactor the WebSocket layer m1nd.warmup({ "context": "Refactoring WebSocket reconnection and pub/sub fan-out", "agent_id": "jimi" })
Before focused work Task preparation Domain priming

⍐ why — Explain path between two nodes

Trace the dependency chain between two nodes in the graph. Answers "Why does A relate to B?" by finding the shortest weighted path and explaining each hop. Critical for understanding hidden coupling and non-obvious dependencies.

Parameters

agent_idstringCalling agent identifier.
Example

Understanding dependencies Tracing influence Architecture analysis

⟁
Lock
Pin subgraph regions, capture baselines, and monitor changes with precise deltas.
5 tools
⟁lock_create— Pin a subgraph + capture baseline

Pin a subgraph region and snapshot its current state as a baseline. Subsequent lock_diff calls report exactly what changed since the snapshot. Supports node, subgraph, path, and query_neighborhood scopes.

Change monitoringPre/post refactor diffCI guardrails
⟁lock_diff— What changed in a locked region?

Compute the structural delta between the lock baseline and the current graph. Returns added/removed nodes and edges, drift score, and risk assessment.

Post-edit verificationRegression detection
⟁lock_watch— Set watcher strategy on a lock

Configure how a lock monitors changes: manual (on demand), on_ingest (triggers after every ingest), or on_learn (triggers after Hebbian updates).

Automated monitoringCI integration
⟁lock_rebase— Re-capture baseline without releasing

Update the baseline of an existing lock to the current graph state — preserving the lock's scope — so future diffs are relative to the new snapshot.

Baseline refreshIterative development
⟁lock_release— Release a lock and free resources

Destroy a lock and release its associated memory and watcher. Always release locks when done to avoid accumulating stale monitors.

CleanupResource management

Trail
Persist, resume, and merge investigation states across sessions. Your work survives restarts.
4 tools
⍐trail_save— Persist investigation state

Serialize the current investigation — nodes visited, hypotheses formed, conclusions reached, open questions — into a durable trail. The trail survives session restarts and can be resumed later with full context restoration.

Multi-session investigationsCollaborationAudit trail
⍐trail_resume— Restore a saved investigation

Load a saved trail and re-inject its activation boosts into the current graph. Detects staleness (>50% missing nodes) automatically. Restores the full context of a previous investigation session.

Session resumptionContext restoration
⍐trail_merge— Combine two investigation trails

Merge two or more trails into a unified investigation. Discovers cross-connections between separate investigation threads. Useful when parallel agents have investigated different regions of the same codebase.

Multi-agent coordinationCross-thread synthesis
⍐trail_list— List saved trails

List all stored investigation trails with metadata: label, status (active/saved/stale), tags, and creation timestamps. Supports filtering by status and tag.

Trail discoverySession management
⍂
Surgical
Read, edit, and apply code changes with full graph awareness. Zero approximation.
5 tools
⍂view— Fast file reader with line numbers

Read any file from the ingested graph with zero traversal overhead. Returns content with line numbers, auto-ingest if not yet indexed. Replaces cat/head/tail for all code navigation. The baseline step before any surgical edit.

File inspectionPre-edit contextQuick audit
⍂surgical_context— Targeted edit context extraction

Return the minimum graph neighbourhood required for a safe, informed edit: file content, symbol definitions, callers, callees, and associated tests. Eliminates the need to load entire modules to understand a single change point.

Before every editCaller/callee analysisTest coverage check
⍂surgical_context_v2— Full workspace snapshot in one call

Superset of surgical_context: returns the primary file's full content PLUS the source of all connected files (callers, callees, tests) in a single call. Designed for multi-file edits where co-change surface area must be fully understood before writing.

Multi-file refactorsAPI surface changes
⍂apply— Write code and trigger re-ingest

Write new content to a file and automatically trigger incremental graph re-ingest so the connectome stays coherent with the code. The graph reflects edits immediately — subsequent queries operate on the updated structure.

Code editsRefactoringAuto-sync graph
⍂apply_batch— Atomic multi-file writes

Write multiple files atomically and trigger a single bulk re-ingest. All-or-nothing semantics: if any file fails, none are written. Use after surgical_context_v2 when editing a primary file and its callers/tests together. Prevents partial-state inconsistencies.

Atomic refactorsMulti-file API changesBatch generation
