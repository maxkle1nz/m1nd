Wiki / L1GHT Protocol

⍂ L1GHT Protocol

A structured markdown format that lets m1nd ingest documentation, specifications, and knowledge bases as typed, weighted knowledge graphs — without writing a line of code.

● adapter: "light"  ·  v0.6.1

What is L1GHT?

L1GHT is m1nd's structured markdown ingest protocol. While the code adapter parses source files (Rust, Python, TypeScript, etc.) into a graph, the light adapter parses markdown documents enriched with a semantic tagging syntax.

The result is the same: a weighted, traversable graph. The difference is the input domain. L1GHT transforms prose — specs, notes, decision records, knowledge bases, investigation trails — into first-class graph citizens that activate, missing, and seek can reason over alongside code.

Key Use Cases

Connect specs to implementations. Find knowledge gaps. Link design decisions to code. Build internal wikis that agents can navigate as graphs, not flat text files.

Protocol Detection

The light adapter auto-detects L1GHT documents. A file qualifies if it satisfies either condition:

  • Contains the header line Protocol: L1GHT/<version>
  • Contains 2 or more inline marker patterns (see below)

Files that don't match are silently skipped. Only .md and .markdown extensions are accepted.

Note

You can mix L1GHT files and non-L1GHT files in the same directory. Only matching files will be ingested. This makes it safe to point the adapter at any docs folder.

YAML Frontmatter

A L1GHT document opens with a --- delimited header block (first 40 lines) containing structured metadata. These fields become Concept nodes in the graph, connected to the file node with typed edges.

Inline Markers

Within the document body, square-bracket markers create additional graph nodes anchored to their line number. Each marker becomes a typed node connected to its parent section (or the file node if no section precedes it).

## Authentication Flow This service handles all session management. The core token validator [⍂ entity: TokenValidator] performs HMAC-SHA256 verification. Current state is [⍐ state: production]. The refresh path [⟁ depends_on: RefreshTokenStore] writes to Redis. Integration tests are [⟁ tests: test_refresh_flow]. [RED blocker: Redis connection pool not yet tuned for peak load] [AMBER warning: Token expiry edge case needs review]

All inline markers produce edges with weight 0.9 and causal_strength 0.7, anchored to their parent ## Section node (or file node if no section exists).

Graph Model

Each L1GHT document produces a sub-graph with the following node hierarchy:

  • File node (NodeType::File) — root, tagged light and namespace:<ns>
  • Metadata nodes (NodeType::Concept) — one per frontmatter field present
  • Dependency nodes (NodeType::Reference) — one per Depends on: entry
  • Next nodes (NodeType::Reference) — one per Next: entry
  • Section nodes (NodeType::Module) — one per ## H2 heading
  • Tag nodes (NodeType::Concept or Process) — one per inline marker

All edges are typed and carry explicit weight and causal_strength values used during spreading activation and impact analysis.

Cross-Domain Fusion

When ingested with mode: "merge", L1GHT nodes share the same graph as code nodes. A ⟁ depends_on: PaymentService marker in a spec will form structural proximity with the actual PaymentService module from a code ingest — enabling cross-domain activation.

Usage

Ingest a L1GHT corpus

// Ingest a docs folder as a light graph (replaces current graph) m1nd.ingest({ "path": "./docs/specs", "adapter": "light", "mode": "replace", "agent_id": "jimi" })

Merge with code graph

// Fuse specs with codebase — enables cross-domain activation m1nd.ingest({ "path": "./src", "adapter": "code", "mode": "replace", "agent_id": "jimi" }) m1nd.ingest({ "path": "./docs", "adapter": "light", "mode": "merge", "namespace": "specs", "agent_id": "jimi" })

Query across both domains

// Activate fires across code AND specs simultaneously m1nd.activate({ "query": "authentication token refresh", "agent_id": "jimi" }) // Find specs with no implementation counterpart m1nd.missing({ "query": "payment flow", "agent_id": "jimi" })

It [⟁ depends_on: StripeClient] for API calls. [⟁ tests: test_charge_happy_path] ## Refunds [⍐ state: beta] — Refund flow is in limited rollout. [AMBER warning: Webhook retry not yet idempotent]
